**Tool Calling: Мини‑исследование архитектуры и пайплайна**

- **Цель:** Понять, как в проекте реализован Tool Calling, где он встраивается в общий пайплайн, из чего состоит архитектура, как устроена фильтрация и как расширять на новые отрасли.
- **Основные источники:** `AGENTS.md`, `CLAUDE.md`, `app/core/tool_calling/README.md`, `app/core/tool_calling/TOOL_CALLING_WORKFLOW.md`, а также код модуля `app/core/tool_calling/` и интеграция в `app/core/engine.py`.

**Место В Пайплайне**
- **Вход:** После семантической маршрутизации и переранжирования файлов маршрутов (routes), а также выбора ключа данных (key selection).
- **Интеграция:** `engine.py` вызывает `ToolService.process_query(...)` на каждый релевантный ключ из выбранного JSON-файла.
- **Выход:** Отфильтрованный `product_list` заменяет исходный список перед финальной генерацией ответа.
- **Fallback:** Если отрасль не поддерживается или шаги tool call не дали результата, возвращаются исходные данные без ошибок в основную цепочку.

**Компоненты Архитектуры**
- **`ToolService` (`service.py`):** Оркестратор.
  - **Вход:** `subsector_id`, текстовый запрос, объект данных (по выбранному ключу), `selected_key`.
  - **Действия:** Подбирает хендлер по `subsector_id`, запускает процесс tool call с защитой от ошибок (graceful fallback).
  - **Выход:** `ToolCallResult` с флагом `success`, отфильтрованными данными и метаданными.
- **`HandlerRegistry` (`registry.py`):** Реестр активных хендлеров по `subsector_id`.
- **`UniversalIndustryHandler` (`universal_handler.py`):** Единый хендлер для всех отраслей.
  - **Особенности:** Динамическая загрузка маппингов отрасли, генерация строгих схем (strict tool schemas), извлечение параметров через LLM, универсальная фильтрация с поддержкой субключей.
- **База типов (`base/`):**
  - `BaseToolHandler`: абстрактный интерфейс хендлера.
  - `ToolCallResult`, `FilterParameters`, `ToolSchema`: типы результатов и схем.
- **Загрузка маппингов:** `IndustryMappingsLoader` внутри `universal_handler.py` маппит `subsector_id → industry_name` и через `importlib` импортирует `keys.py`, `enums.py`, `universal.py` отрасли.

**Структура Маппингов (per‑индустрия)**
- **Папки:** `app/core/tool_calling/<industry>/mappings/`
  - `keys.py`: описания полей (ключей) продуктов и влияние фильтрации.
  - `enums.py`: перечисления значений, часто с вложенными «категориями» (субключи).
  - `universal.py`: функции и паттерны для сопоставления текстов, иногда с числовыми проверками.
- **Ключевые объекты:**
  - `<INDUSTRY>_KEYS_MAPPING`: `{ file.json: { product_keys: {...} } }`
  - `<INDUSTRY>_ENUM_MAPPING`: `{ file.json: { field: { category: [values] } } }`
  - `KEY_TO_FILE_MAPPING` (опционально): сопоставляет `selected_key` сложной структуре/файлу.

**Генерация Tool‑схем**
- **Алгоритм:**
  - Определение имени файла по `selected_key` с учетом `KEY_TO_FILE_MAPPING`.
  - Загрузка `product_keys` (описание полей) и соответствующих `enum` значений (включая субключи, где применимо).
  - Сборка строгой схемы для LLM: 
    - `strict: true`, `additionalProperties: false`.
    - `properties`: для каждого поля тип `["string", "null"]`, `description` из `keys.py`, `enum` из объединения значений (значения + `null`).
    - `required`: все объявленные ключи.
    - Имя функции в схеме: `filter_{industry}_products`.
- **Результат:** Массив схем (на текущем коде — одна строгая схема на файл/ключ).

**Вызов LLM И Извлечение Параметров**
- **Промпт:** Строгий системный промпт (Milk‑подход): выбрать ОДИН инструмент и по одному значению на параметр, допускается `null`.
- **Вызов:** `llm_service.chat.completions.create(..., tools=schemas, tool_choice="auto")`.
- **Парсинг:** Берется первый `tool_call`, из аргументов JSON удаляются пустые/`null` значения. Результат — `FilterParameters`.
- **Ошибки:** Если инструмент не вызван или парсинг не удался — возвращаются пустые параметры, что ведет к безопасному возврату исходных данных.

**Фильтрация Данных**
- **Извлечение `product_list`:** Универсальный поиск в корне и субключах; поддержка строкового JSON с `json.loads`.
- **Механика фильтрации:** Последовательное применение фильтров по каждому полю из `FilterParameters`.
  - Для каждого значения берутся «универсальные паттерны» из `mappings.universal.get_universal_patterns(value)`.
  - Совпадение — простое подстрочное соответствие любого паттерна в текстовом представлении продукта.
- **Числовая логика:** В отраслевых `universal.py` (например, HoReCa, Milk) существуют функции для диапазонов (калорийность, дозировки и т.п.). В текущем универсальном хендлере используется упрощенный текстовый матч через паттерны. Если требуется точная числовая фильтрация, её можно интегрировать, используя отраслевые хелперы.
- **Результат:** Обновление `product_list` в исходной структуре данных; пустые списки детектируются и ведут к fallback.

**Fallback И Надежность**
- **Уровни fallback:**
  - `ToolService`: отсутствие хендлера для `subsector_id` и любые исключения → возврат исходных данных.
  - `UniversalIndustryHandler`: пустые схемы, отсутствие параметров, пустой результат фильтрации, любые ошибки → возврат исходных данных.
- **Формат результата:** `ToolCallResult` с полями `success`, `filtered_data`, `applied_filters`, `error_message?`, `metadata?`.

**Интеграция В Engine**
- **Файл:** `app/core/engine.py`.
- **Поток:** `rerank_routes → read_and_merge → select_relevant_keys → ToolService.process_query → generate_final_answer`.
- **Контекст:** В финальную генерацию передается объединенный JSON только с обработанными/отфильтрованными `product_list` по выбранным ключам.

**Поддерживаемые Отрасли И ID**
- **В коде:** `UniversalIndustryHandler` инициализируется для `"01"` (HoReCa), `"04"` (Milk), `"00"` (Selo); регистрация в `ToolService._register_handlers`.
- **В документах:** В `CLAUDE.md` упоминается Milk как `"02"` — это расхождение с актуальным кодом (следует унифицировать).

**Расширение На Новые Отрасли**
- **Создать маппинги:** `app/core/tool_calling/<industry>/mappings/{keys.py,enums.py,universal.py}`.
  - Экспортировать: `<INDUSTRY>_KEYS_MAPPING`, `<INDUSTRY>_ENUM_MAPPING`, при необходимости `KEY_TO_FILE_MAPPING` и `get_universal_patterns`.
- **Подключить в коде:**
  - Добавить соответствие `subsector_id → industry_name` в `IndustryMappingsLoader.INDUSTRY_MAPPING`.
  - Зарегистрировать хендлер в `ToolService._register_handlers()` через `UniversalIndustryHandler(<subsector_id>, llm_service)`.
- **Время:** По документации — 30–60 минут за счет унифицированной архитектуры.

**Сильные Стороны Подхода**
- **Единый хендлер:** Унификация логики и снижение дублирования.
- **Динамические схемы:** Автоматическая генерация строгих инструментов из маппингов отрасли.
- **Гибкая фильтрация:** Изолированные отраслевые паттерны в `universal.py` + поддержка субключей и сложных JSON.
- **Надежность:** Многоуровневый graceful fallback, не ломает основной пайплайн.

**Наблюдения И Возможные Улучшения**
- **ID Milk:** В коде используется `"04"`, в документах встречается `"02"`. Рекомендуется унифицировать значения и документы.
- **Числовая фильтрация:** В отраслевых `universal.py` есть числовые/regex‑хелперы (например, калорийность), но универсальный хендлер сейчас применяет только текстовые паттерны. Стоит добавить маршрутизацию «по полю» к соответствующим числовым проверкам.
- **Строгость схем:** Сейчас `required` заполняется «всеми ключами» — это повышает вероятность `null` при неполном запросе. Альтернатива — динамически определять `required` на основе запроса/приоритета полей.
- **Тесты:** Добавить юнит‑тесты на генерацию схем по разным ключам, извлечение параметров (мока LLM), фильтрацию по текстовым и числовым критериям.

**Краткий Поток Данных (с Tool Calling)**
- Запрос → Семантическая маршрутизация → Переранжирование → Выбор ключа → Генерация строгой tool‑схемы → Вызов LLM с инструментом → Применение фильтров → Обновленный `product_list` → Финальная генерация ответа.
