# ЗАДАЧИ ПРОЕКТА SZSB_SALES

## ТЕКУЩАЯ ЗАДАЧА: Создание логики выбора tools LLM для отрасли HoReCa

### ОПИСАНИЕ ЗАДАЧИ
Необходимо создать экспериментальную логику выбора и применения инструментов (tools) для LLM, которая будет встроена в пайплайн обработки запросов между этапами `select_relevant_keys` и `generate_final_answer`. Функциональность активируется исключительно для отрасли "01": "horeca".

### СИСТЕМНЫЙ АНАЛИЗ

#### ЦЕЛЬ
Внедрить промежуточный этап Tool Calling в существующий пайплайн для улучшения точности обработки запросов в сфере HoReCa через использование специализированных инструментов LLM.

#### КОНТЕКСТ
- Текущий пайплайн: Query → Semantic Routing → Reranking → Key Selection → Context Preparation → Final Generation → Response
- Целевой пайплайн для HoReCa: Query → Semantic Routing → Reranking → Key Selection → **Tool Calling** → Context Preparation → Final Generation → Response

#### ТРЕБОВАНИЯ
1. **Функциональные требования:**
   - Активация только для subsector_id = "01" (horeca)
   - Интеграция между `select_relevant_keys` и `generate_final_answer`
   - Совместимость с существующим пайплайном
   - Обратная совместимость для других отраслей

2. **Нефункциональные требования:**
   - Логирование всех операций
   - Обработка ошибок с fallback
   - Модульная архитектура
   - Качественная документация кода

### РЕАЛИЗОВАННЫЕ КОМПОНЕНТЫ

#### 1. **horeca_keys_mapping.py**
Полный анализ структуры данных всех 9 файлов HoReCa с описаниями ключей для генерации tool схем.

#### 2. **horeca_enum_mapping.py**
Enum значения для каждого ключа в каждом файле HoReCa, используемые LLM для выбора фильтров.

#### 3. **universal_enum_mapping.py**
Универсальный промежуточный маппинг enum → реальные данные для всех файлов HoReCa с поддержкой:
- Regex для числовых значений (калории, белки, жиры, углеводы)
- Текстовые паттерны для брендов, типов упаковки, условий хранения
- Специальные функции проверки диапазонов

#### 4. **horeca_filter.py**
Умная функция фильтрации с использованием pandas DataFrame и промежуточного маппинга.

#### 5. **horeca_tool_service.py**
Основной класс HoReCaToolService с полным циклом обработки:
- Динамическая генерация OpenAI tool схем
- Интеграция с LLM для выбора инструментов
- Фильтрация продуктов по выбранным критериям
- Обработка ошибок и fallback механизмы

#### 6. **test_horeca_tool_calling.py**
Изолированный тестовый файл с MockLLMService для валидации всей логики без зависимостей.

#### 7. **Интеграция в engine.py**
Добавлена условная логика Tool Calling для HoReCa между select_relevant_keys и generate_final_answer.

### АРХИТЕКТУРА РЕШЕНИЯ

#### Обычный флоу (все подсекторы кроме HoReCa):
```
Query → Semantic Routing → Reranking → Key Selection → Context Preparation → Final Generation → Response
```

#### HoReCa флоу (с Tool Calling):
```
Query → Semantic Routing → Reranking → Key Selection → **Tool Calling** → Context Preparation → Final Generation → Response
```

#### Компоненты Tool Calling:
1. **generate_tool_schema()** - создает динамическую схему на основе реальных данных
2. **call_llm_with_tool()** - вызывает LLM для выбора подходящего инструмента
3. **filter_products()** - применяет фильтрацию по выбранным критериям
4. **process_horeca_query()** - выполняет полный цикл обработки

### ОСОБЕННОСТИ РЕАЛИЗАЦИИ

#### 1. **Динамические enum'ы**
Схемы инструментов генерируются на основе реальных данных файла, создавая enum'ы с доступными значениями.

#### 2. **Промежуточный маппинг**
Решает проблему несоответствия между "чистыми" enum'ами и реальными данными через паттерны и regex.

#### 3. **Regex для числовых значений**
Автоматическое извлечение и проверка калорийности, белков, жиров, углеводов из текста КБЖУ.

#### 4. **Graceful degradation**
При ошибках в Tool Calling система возвращает исходные данные без фильтрации.

#### 5. **Обратная совместимость**
Другие подсекторы работают без изменений, Tool Calling активируется только для HoReCa.
