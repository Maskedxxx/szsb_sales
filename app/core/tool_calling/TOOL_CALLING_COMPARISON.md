# Сравнение Tool Calling Workflow: HoReCa vs. Milk

## Введение

Этот документ описывает и сравнивает пайплайны модуля Tool Calling для двух отраслей: **HoReCa** (`subsector_id="01"`) и **Milk** (`subsector_id="04"`). Хотя общая архитектура модуля едина, конкретные реализации обработчиков (`Handler`) имеют ключевые различия, обусловленные спецификой и структурой данных каждой отрасли.

---

## ♠️ Общие элементы пайплайна (Совпадения)

Оба обработчика (`HoReCaHandler` и `MilkHandler`) работают в рамках единой архитектуры и разделяют следующие этапы и компоненты:

1.  **Единая точка входа**: Весь процесс для обеих отраслей инициируется через главный сервис `ToolService`.
2.  **Реестр обработчиков**: `ToolService` использует `HandlerRegistry` для определения и получения нужного обработчика (`HoReCaHandler` или `MilkHandler`) на основе `subsector_id`.
3.  **Абстрактная база**: Оба класса-обработчика являются наследниками `BaseToolHandler` и реализуют его "контракт" — методы `generate_schema` и `filter_data`.
4.  **Общий воркфлоу**: Последовательность шагов идентична на высоком уровне:
    *   Проверка поддержки отрасли (`is_supported`).
    *   Получение нужного `Handler`.
    *   Вызов `handler.process()`, который оркестрирует внутреннюю логику.
    *   Генерация динамической схемы на основе данных.
    *   Вызов LLM для извлечения параметров фильтрации.
    *   Применение "умной" фильтрации к данным.
    *   Возврат результата в виде стандартизированного объекта `ToolCallResult`.
5.  **Умная фильтрация**: Оба обработчика используют схожие механизмы в своих модулях `data_filter.py` для сопоставления критериев с данными, включая:
    *   Поиск по текстовым синонимам и паттернам (`get_universal_patterns`).
    *   Использование регулярных выражений и специальных функций для числовых данных (калорийность, дозировка).
6.  **Отказоустойчивость (Graceful Fallback)**: Если на любом из этапов (генерация схемы, вызов LLM, фильтрация) происходит ошибка, система не падает, а возвращает исходные, нефильтрованные данные, обеспечивая стабильность всего пайплайна.

---

## ♦️ Ключевые различия в реализации (HoReCa vs. Milk)

Несмотря на общую структуру, детали реализации кардинально отличаются, что продиктовано сложностью и структурой данных в каждой отрасли.

### 1. Структура данных и маппинги

-   **HoReCa (Простая структура)**:
    *   **Источник данных**: Простая и плоская структура. `selected_key` напрямую соответствует имени файла (например, `ready_sauces` → `ready_sauces.json`).
    *   **Список продуктов**: Всегда находится по одному и тому же пути: `data["product_list"]`.
    *   **Маппинги**: `HORECA_KEYS_MAPPING` и `HORECA_ENUM_MAPPING` имеют простую структуру "файл -> ключи" и "файл -> enums".

-   **Milk (Сложная структура с субключами)**:
    *   **Источник данных**: Поддерживает сложные, вложенные JSON-структуры. Один файл может содержать несколько списков продуктов под разными **субключами** (например, `delar_flavor_collection.json` содержит `gastronomic_flavors` и `sweet_flavors`).
    *   **Список продуктов**: Может находиться как в корне (`data["product_list"]`), так и во вложенных объектах (`data["dyes"]["product_list"]`). Логика фильтрации должна уметь находить его.
    *   **Маппинги**: `MILK_ENUM_MAPPING` имеет иерархическую структуру, где enum'ы могут быть определены для каждого **субключа** внутри файла. Требуется специальная логика для поиска файла по ключу (`_find_file_for_key`).

### 2. Генерация OpenAI Tool Схемы

Это самое главное и принципиальное различие между двумя обработчиками.

-   **HoReCa (Стандартный режим)**:
    *   **Обязательность полей**: Все поля в схеме являются **опциональными**. `required: []`.
    *   **Типы данных**: Используется простой тип `"type": "string"`.
    *   **Null значения**: `null` **не** добавляется в `enum` и типы данных.
    *   **Итог**: Генерируется простая и гибкая схема, где LLM может передавать только те параметры, которые нашел в запросе.

-   **Milk (Режим `strict: true`)**:
    *   **Обязательность полей**: Все поля в схеме являются **обязательными**. `required: ["name", "dosage", "color", ...]`. Это требование `strict` режима OpenAI.
    *   **Типы данных**: Чтобы обязательные поля могли быть пропущены, используется составной тип: `"type": ["string", "null"]`.
    *   **Null значения**: `null` (в коде `None`) **обязательно** добавляется в список `enum` для каждого поля.
    *   **Итог**: Генерируется очень строгая схема. LLM **обязан** для каждого поля вернуть либо одно из значений enum, либо `null`.

### 3. Системный промпт для LLM

Различия в схемах напрямую влияют на инструкции, которые даются LLM.

-   **HoReCa**:
    *   Промпт более мягкий: *"Важно: все параметры являются опциональными. Передавай только те, которые явно указаны в запросе."*

-   **Milk**:
    *   Промпт гораздо более строгий, чтобы соответствовать `strict` схеме: *"ВАЖНО: Ты ДОЛЖЕН вызвать инструмент ТОЛЬКО ОДИН РАЗ... Обязательно выбери только ОДНО значение для каждого параметра или null если параметр не нужен."*

### 4. Логика извлечения данных для фильтрации

-   **HoReCa**:
    *   Логика проста: список продуктов всегда извлекается из `data["product_list"]`.

-   **Milk**:
    *   Логика сложнее: система сначала ищет `product_list` в корне объекта `data`. Если не находит, она итерируется по всем вложенным словарям в поиске ключа `product_list`, чтобы поддержать работу с субключами.

## 📊 Итоговая таблица различий

| Аспект | HoReCa | Milk |
| :--- | :--- | :--- |
| **Структура данных** | Простая, плоская. `selected_key` = `имя_файла.json`. | Сложная, вложенная. Поддержка **субключей** в файлах. |
| **Поиск `product_list`** | Всегда в `data["product_list"]`. | Динамический поиск: в корне или во вложенных объектах. |
| **Генерация схемы** | Стандартный режим. | **OpenAI Strict Mode** (`strict: true`). |
| **Обязательность полей**| Все поля опциональны (`required: []`). | Все поля обязательны (`required: ["все", "поля"]`). |
| **Типы полей в схеме** | `"type": "string"` | `"type": ["string", "null"]` |
| **`null` в `enum`** | Не добавляется. | **Обязательно** добавляется. |
| **Системный промпт LLM**| Мягкий (параметры опциональны). | Строгий (выбери значение или `null` для **каждого** поля). |
